FormValidator

A smart, easy-to-use, and fast JavaScript form validator.

All it needs is jQuery (≥ 1.7).

JavaScript free form validation
Validate any element without needing to write any JavaScript (similar to Bootstrap's modal, for example).
Simple type-based validation
Validate forms simply by using some HTML attributes.
Constraints
Easily define constraints on form fields in addition to their validation kind.
Any constraint can be added to any field.
Required and optional fields
Define what fields are optional by using a predefined attribute on the according elements.
Easy theming
Since CSS classes are applied to invalid elements, creating custom themes is easy.
The FormValidator ships with some themes that flawlessly integrate into a page using Bootstrap.
Dependencies
Define dependencies between form fields.
That means that a dependent field can only become valid when its required fields (dependencies) are also valid. Event handlers can automatically be triggered, for example, such a field's visibility could be toggled (see example 3).
Detailed errors
Invalid fields produce user friendly errors in various formats.
And it's possible to customize the degree of detail!
Real time validation
Validate form fields on the fly.
Form progress
Find out how much a user has filled in a form in real time.
i18n
Each form validator generates the error messages depending on its locale property.
They can be customized, more can be added. English and German are predefined.
Customization
Lots of other options can be used for a form validation by using the JavaScript API.

Include the FormValidator

include it after jQuery (it requires jQuery 1.7+ upon initialization!)
            <script type="text/javascript" src="/path/to/FormValidator(.min).js"></script>
The development version (FormValidator.js) throws more exceptions and prints errors. Therefore, it is recommended to be used for more convenient development. ;)

(3) Using JavaScript, printing error messages, dependencies

The form

Locale:
Keep in mind that FormValidator comes with some predefined preprocessors.
For example, when choosing 'de' the string 123.15,45 will be parsed as 12315.45.

The JavaScript

            $("#example3").click(function() {
                var form = $("#example3").closest("form"),
                    error, div, text, i, j, target, hide_field;
                var validator = FormValidator.new(form, {
                    error_target_getter: function(element, type) {
                        return element.closest(".row").find(".error_state");
                    },
                    dependency_change_action: FormValidator.DEPENDENCY_CHANGE_ACTIONS.SLIDE,
                    locale: form.find(".example3.locale option:selected").val()
                });
                var errors = validator.validate();

                form.find(".error_message").empty();
                for (i = 0; i < errors.length; i++) {
                    error = errors[i];
                    div = error.element.closest(".row").find(".error_message");
                    text = div.text();
                    div.text(text + (text.length > 0 ? "; " : "") + error.message);
                }
                return true;
            });

The following validators come with the FormValidator. The list shows what values will be interpreted as valid.
  • checkbox
    a checked checkbox
  • email
    <ANYTHING+>@<ANYTHING+>.<ANYTHING_WITHOUT_DOT+> where <ANYTHING> must not contain an @
  • integer
    any integer x ∈ ℤ (that JavaScript can represent),
    see constraints for defining a range
  • number
    any number x ∈ ℝ (that JavaScript can represent),
    see constraints for defining a range
  • phone
    any combination of 0123456789 +-/()
  • radio
    a radio button of the element's group of radio buttons (= all with the same name attribute) must be checked
  • select
    the selected item's value is not ""
  • text
    anything longer than 0 characters,
    see constraints for defining a minimum or maximum length

The following constraints come with the FormValidator

data-fv-blacklist
String
Define a set of characters of which none may occur in the validation value of the field.
There are no default options for this constraint.
data-fv-max
Number data-fv-include-max
Define the maximum value for an element.
By default this value is the greatest valid value. To exclude the maximum see the option data-fv-include-max.
data-fv-max-length
Number
Define the maximum length for the validation value of an element.
This value is the greatest valid value.
There are no default options for this constraint.
data-fv-min
Number data-fv-include-min
Define the minimum value for an element.
By default this is be the smallest valid value. To exclude the minimum see the option data-fv-include-min.
data-fv-min-length
Number
Define the minimum length for the validation value of an element.
This value is the smallest valid value.
There are no default options for this constraint.
data-fv-regex
String data-fv-regex-flags
Define a regular expression that the element's validation value must not match.
data-fv-whitelist
String
Define a set of characters that the element's validation value must contain.
There are no default options for this constraint.

Note

If you define a constraint to a validator so that the combination does not make sense FormValidator won't throw or report anything nor try anything fancy. I decided to keep the code simple (and as lightweight as possible) and make the user take care of correctness of validation definitions.
FormValidator alwasy assumes syntactically and logically correct input from the programmer.

Dependencies can be used on any form field - let's pick one and call it F. The dependencies define what other form fields need to be valid before F can be valid (no matter the content).

It is possible to change F whenever its dependencies are "valid enough". With "valid enough" I mean that either

  • all of F's dependencies are fulfilled (onDependenciesValid) or
  • any of F's dependencies is fulfilled (onDependenciesInvalid).
Whether all or any is evaluated can be defined with the data-fv-dependency-mode attribute (either of ["all", "any"], default is all).

The change action can be anything if you use JavaScript with the dependency_change_action option.
However, usually you want to toggle the state of F on each event. So there is a list of predefined toggling changes. The naming for the actions is based on "do something whenever the dependencies are fulfilled - otherwise reverse whatever has been done before". Here we go:

  • DEPENDENCY_CHANGE_ACTIONS.FADE *
    Fade the element in (on onDependenciesValid) or out (on onDependenciesInvalid).
    See jQuery's fadeIn and fadeOut.
  • DEPENDENCY_CHANGE_ACTIONS.OPACITY *
    Same as fade but the display style is not changed.
  • DEPENDENCY_CHANGE_ACTIONS.SHOW *
    See jQuery's show and hide.
  • DEPENDENCY_CHANGE_ACTIONS.SLIDE *
    See jQuery's slideDown and slideUp.
  • DEPENDENCY_CHANGE_ACTIONS.ENABLE
    Change the disabled attribute (disabled on onDependenciesInvalid).
  • DEPENDENCY_CHANGE_ACTIONS.DISPLAY
    The same as show but without animnation (changes the display style).
* The duration used for the animation is jQuery's default value (currently 400 ms). It is hardcoded though and will therefore not change if jQuery changes it.

Cycle detection

There is none!
So you must make sure you do not define cyclic dependencies. Since dependencies are defined as a list of targets you could use a selector to find a dependency's element (or multiple). In that case you should be careful to match only the wanted elements.
Using the data-fv-name attribute should be easier and safer to use and is therefore recommended.

Error handling

build_mode
String
Defines how error message are created.
Must be either the options defined in FormValidator.BUILD_MODES.
Default is FormValidator.BUILD_MODES.DEFAULT.
See configuration for more details.
constraint_validators
Object
Can be used to add or override constraint validators to the predefined set.
See constraints for more details.
dependency_change_action
String
Defines what effect is applied to the data-fv-dependency-action-targets.
Must be either the options defined in FormValidator.DEPENDENCY_CHANGE_ACTIONS.
Default is FormValidator.DEPENDENCY_CHANGE_ACTIONS.DEFAULT.
See configuration for more details.
dependency_error_classes
String (CSS class names, semicolon separated)
Defines what CSS classes will be applied to the error target(s) of the fields the current field depends on (if any of the dependencies are invalid).
Default is "fv-invalid-dependency".
Take a look at example 3 on the demos tab to see how dependencies work.
error_classes
String (CSS class names, semicolon separated)
Defines what CSS classes will be applied to the error target(s) of an invalid field.
Default is "fv-invalid".
error_mode
FormValidator.ERROR_MODES (String)
Defines how error messages are generated in general.
Must be either the options defined in FormValidator.ERROR_MODES.
Default is FormValidator.ERROR_MODES.DEFAULT.
See configuration for more details.
error_output_mode
FormValidator.ERROR_OUTPUT_MODES (String)
Defines how error messages are shown. Can be either of these values:
  • FormValidator.ERROR_OUTPUT_MODES.BELOW
    Prints an error message below the according validated form field.
  • FormValidator.ERROR_OUTPUT_MODES.BOOTSTRAP_POPOVER
    Puts an error message into a bootstrap popover that shows when hovering over the according form field.
    Of course, this only works when Boostrap is included.
  • FormValidator.ERROR_OUTPUT_MODES.NONE
    The DOM will not be modified. The validate function returns an Array of Objects where each of those objects looks like this:
                                {
                                element: validated_field,
                                message: error_message,
                                required: validated_field.attr("data-fv-optional") !== "true",
                                type: validation_type,
                                validator: validator_function,
                                value: postprocessed_validated_field_value || preprocessed_validated_field_value || validated_field_value
                            }
  • FormValidator.ERROR_OUTPUT_MODES.DEFAULT (== FormValidator.ERROR_OUTPUT_MODES.NONE) (this value is the default obviously)
error_target_getter
Function(String validation_type, jQuery element, Number index) -> jQuery | String
Can be used to define the error target(s) for a validated field.
If a jQuery set is returned all the contained elements will be the targets.
If a String is returned it is parsed the same as if it was set using the data-fv-error-targets attribute. See target finding.
process_errors
Function(Array (of Object) grouped_errors) -> void
This function can be used to modify the validation errors in place before they are passed anywhere else (but after the error message have (potentially) been created).
success_classes
String (CSS class names, space separated)
Defines what CSS classes will be applied to the error target(s) of a valid field.
Default is "fv-valid".

Validation handling

field_getter
Function(jQuery form) -> jQuery
Can be used to define the set of form fields that are relevant to the FormValidator.
group
Function(jQuery fields) -> Array (of Array of jQuery)
Creates groups from all form fields. Can be used for advanced progress counting.
See methods (get_progress method) for more details.
locale
Define what locale is used. This property is relevant for:
  • creating error message
  • pre- and postprocessing values
optional_field_getter
Function(jQuery fields) -> jQuery
Can be used to define the subset of all fields (see field_getter) that contains all the fields that are optional. Those fields will only be validated if they are non-empty.
postprocessors
Object. It looks like this: 
                    {
                        validation_type1: POSTPROCESSOR_1,
                        ...,
                        validation_typeN: POSTPROCESSOR_N
                    }
where POSTPROCESSOR_N is a Function(String value, jQuery element, String locale) -> String.
Can be used to override or define postprocessors. They are used modify the form field's value after the validation happened. The value is either the input's original or preprocessed value.
For example, a value could be auto-corrected i.e. the numeric string " 1.34 " would become "1.34" in the according field (when valid).
preprocessors
Object. It looks like this: 
                    {
                        validation_type1: PREPROCESSOR_1,
                        ...,
                        validation_typeN: PREPROCESSOR_N
                    }
where PREPROCESSOR_N is a Function(String value, jQuery element, String locale) -> String.
Can be used to override or define preprocessors. They are used modify the form field's value before the validation happens.
The predefined preprocessors currently exist for number and integer. Currently, for both types the comma will be interpreted as decimal points when the locale de is used.
required_field_getter
Function(jQuery fields) -> jQuery
Can be used to define the subset of all fields (see field_getter) that contains all the fields that are required. Those fields will always be validated.
validation_options
Specify options for the validation:
  • Boolean all: if true force validation on all fields (even if they are declared optional)
  • Boolean apply_error_styles: if true add error_classes to and remove them from invalid elements' error targets (otherwise the targets won't be changed)
validators
Set of validators that correspond to a validation type (data-fv-validate).
They can be used to override existing validators or define new ones.
The predefined validators are:
  • checkbox
  • email
  • integer
  • number
  • phone
  • radio
  • select
  • text

Instance methods

A FormValidator object has the following instance methods. Most of the functionality can be used equally by passing options to the FormValidator's constructor which is recommended for most use cases.
Function cache() -> FormValidator chainable
Cache all kinds of stuff that is needed for validation. Some caching will be done lazily (even after calling this method). Using this method makes sense for real time validation because that way the form data can be cached e.g. right after the has loaded so all validations will be equally fast.
Function get_progress(Object options={mode: PROGRESS_MODES.DEFAULT, recache: false}) -> Array (of Number | Object)
This method calculates how much the form has been filled in. The returned array contains the progress for each group (defined with the group option or defined using the data-fv-group attribute). The array also has a property called average that equals the overall form progress.
Depending on the mode the array elements are either Numbers or Objects. The PROGRESS_MODES.PERCENTAGE version uses decimals between 0 and 1, the objects in the PROGRESS_MODES.ABSOLUTE look like {count: x, total: y}.

This is how the calculation works. It depends on the 3 kinds of groups:
  1. containing only required fields
    progress = valid_fields / num_fields
  2. containing only optional fields
    progress = either 1 if at least 1 field is valid, 0 otherwise
  3. containing both kinds of fields
    progress = valid_required_fields / num_required_fields
    (so this is essentially equal to 1.)
Function validate(Object validation_options={all: false, apply_error_classes: true, focus_invalid: true, messages: true, stop_on_error: true, recache: false}) -> Array (of Object)
Validates the form and returns a list of errors. Each error looks like this: 
                        {
                            element: validated_element (jQuery),
                            errors: list_of_errors (Array of Object),
                            message: error_message (String)
                        }
Each of the error objects looks like: 
                        {
                            element: validated_element (jQuery),
                            error_message_type: error_message_type (String),
                            phase: validation_phase (String),
                            required: is_required (Boolean),
                            type: validation_type (String),
                            value: value (String)
                            // if dependency error
                            mode: dependency_mode (String),
                            // if constraint error
                            <constraint_name>: constraint_value (String),
                            options: constraint_options (Object)
                        }

The validation_options has the following properties (for default values see function signature):
  • Boolean all: if true force validation on all fields (even if they are declared optional)
  • Boolean apply_error_classes: if true adds error_classes to invalid and removes them from valid elements' error targets, if false nothing will be changed
  • Boolean focus_invalid: if true focus the first invalid element (otherwise the focus won't be changed)
  • Boolean message: if true error messages will be created
  • Boolean stop_on_error: if true the validation of an element will be stopped as early as possible - after the first error occured
  • Boolean recache: if true the cache will be emptied and refilled

List of configuration constants

The following configuration constants are attached to the FormValidator class.
If it has a DEFAULT property it points to the according constant marked with default.
Most of them simulate enum of other languages but some of them can be used as variables. Those are marked with variable.

BUILD_MODES
Defines how error message are created.
  • ENUMERATE default
    Combines multiple errors of the same validation phase to an enumeration to be as close to the natural language as possible. Another advantage is that the message will be as short as possible which is preferable especially for mobile devices.
  • LIST
    The messages are generated like SENTENCE but are formatted as HTML ul list.
  • SENTENCE
    Unlike ENUMERATE each error is put into a single sentence. This is more verbose but can be formatted more easily.
DEPENDENCY_CHANGE_ACTIONS
Defines how a change of the validation state of a dependency is visualized.
For the details see dependencies.
  • DISPLAY
  • ENABLE
  • FADE
  • NONE default
    No animation.
  • OPACITY
  • SHOW
  • SLIDE
DEPENDENCY_CHANGE_ACTION_DURATION variable
Defines the global duration for all dependency change actions. This value can be overriden for each field using the data-fv-dependency-action-duration attribute.
The value of this variable is 400 ms.
ERROR_MODES
Defines the degree of detail for error messages.
  • NORMAL default
    The message is created based on the error_message_type returned by the validator.
  • SIMPLE
    The message is based on the base type of the error_message_type returned by the validator. E.g. invalid e-mail values will always produce the same message (like 'invalid e-mail address') instaed of an exact message (like 'missing @ symbol').
ERROR_OUTPUT_MODES
Defines how error message are visualized.
  • BELOW
    Creates a container <div class="fv-error-message" /> containing the message right behind the element (in the DOM).
  • NONE default
    Well...nothing is put into the DOM. That way you can take of everything yourself.
  • POPOVER (required Bootstrap)
    Shows a Bootstrap popover on focus, hides it on blur and toggles it on click.
  • TOOLTIP (required Bootstrap)
    Show a Bootstrap tooltip on hover.
PROGRESS_MODES
Defines how each element of the array looks like.
  • PERCENTAGE default
    Each element is a Number.
  • ABSOLUTE
    Each element is an Object of the form {count: Number, total: Number}.
VALIDATION_PHASES
Defines all of the validation phases. Those constants can be used to group error objects without needing to know the constant value.
  • DEPENDENCIES
  • VALUE
  • CONSTRAINTS

Configuring internal parameters (advanced)

If you want to change some of FormValidator's behavior without modifying its source code directly you can use the FormValidator.configure method:

Function FormValidator.configure(Function callback) -> FormValidatorClass chainable
The callback function parameter has the signature Function callback(Object configurations).
The configurations object is a set of internal variables that influence the behavior of the FormValidator and are made accessible only in that way.
This is the list of internal variables:
  • constraint_validator_groups (Array (of Array of String))
    Defines which constraint validators are logically connected (and thus can potentially be combined to a new error message. For example, if you wanted to have a special error message in case the 2 constraint validators blacklist and whitelist are violated you would add an according group with
    constraint_validator_groups.push(["blacklist", "whitelist"]); Now you would add a new entry constraint_blacklist_whitelist in the locales variables and it will be used.
    The groups defined in that array must be disjoint.
  • constraint_validator_options_in_locale_key
  • constraint_validator_options
  • locale_message_builders
  • message_builders
  • message_data_creators

data-fv-blacklist
String
Define a set of characters that the element's validation value must not contain.
For more details see constraints.
data-fv-dependency-action-duration
Number (in milliseconds)
Can be used to override the default duration.
Default is FormValidator.DEPENDENCY_CHANGE_ACTION_DURATION.
See configuration for more details.
data-fv-dependency-action-targets
String (targets, space separated)
Can be used to customize which CSS classes will be applied to the error target(s) of a field with dependencies (if any of the dependencies are invalid).

Target finding

A list of targets is defined as semicolon separated String.

Each list element E is parsed like so (for a FormValidator F):

  1. try to find an element in the form of F where the attribute data-fv-name equals E
  2. otherwise E is interpreted as selector which will be looked for in the path to F (using jQuery's closest()), this makes it easy to affect container elements of E
  3. otherwise E is interpreted as selector which will be looked for in F
  4. otherwise E is interpreted as selector which will be looked for in the entire document

Note: you cannot use the ; character in the selector because it's used as the target delimiter.
