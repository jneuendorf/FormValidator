FormValidator

A smart, easy-to-use, and fast JavaScript form validator.

All it needs is jQuery (≥ 1.7).

JavaScript free form validation
Validate any element without needing to write any JavaScript (similar to Bootstrap's modal, for example).
Simple type-based validation
Validate forms simply by using some HTML attributes.
Required and optional fields
Define what fields are optional by using a predefined attribute on the according elements.
Easy theming
Since CSS classes are applied to invalid elements, creating custom themes is easy.
Dependencies
Define dependencies between form fields.
That means that a dependent field can only become valid when its required fields (dependencies) are also valid. Event handlers can automatically be triggered, for example, such a field's visibility could be toggled (see example 3).
Detailed errors
Invalid fields produce user friendly errors.
And it's possible to customize the degree of detail!
Real time validation
Validate form fields on the fly.
Form progress
Find out how much a user has filled in a form in real time.
i18n
Each form validator generates the error messages depending on its locale property.
They can be customized, more can be added. English and German are predefined.
Customization
Lots of other options can be used for a form validation by using the JavaScript API.

Include the FormValidator

include it after jQuery (it requires jQuery 1.7+ upon initialization!)
            <script type="text/javascript" src="/path/to/FormValidator(.min).js"></script>
The development version (FormValidator.js) throws more exceptions and prints errors. Therefore, it is recommended to be used for more convenient development. ;)

Create a form and set some attributes

For more information take a look at the demos and the attribute index. 
            <form id="my_form" data-fv-error-classes="invalid" data-fv-error-targets="self">

                    <label>Email address</label>
                    <input type="text" data-fv-validate="email" />
                </div>
                <div>

                </div>

                    <input type="text" data-fv-validate="number" data-fv-optional="true" />

(3) Using JavaScript, printing error messages, dependencies

The form

Locale:
Keep in mind that FormValidator comes with some predefined preprocessors.
For example, when choosing 'de' the string 123.15,45 will be parsed as 12315.45.

The JavaScript

            $("#example3").click(function() {
                var form = $("#example3").closest("form"),
                    error, div, text, i, j, target, hide_field;
                var validator = FormValidator.new(form, {
                    error_target_getter: function(element, type) {
                        return element.closest(".row").find(".error_state");
                    },
                    locale: form.find(".example3.locale option:selected").val()
                });
                var errors = validator.validate();

                form.find(".error_message").empty();
                for (i = 0; i < errors.length; i++) {
                    error = errors[i];

                    if ((target = error.element.attr("data-fv-depends-on")) != null) {
                        hide_field = false;
                        for (j = 0; j < errors.length; j++) {
                            if (errors[j].element.attr("data-fv-name") === target) {
                                hide_field = true;
                                break;
                            }
                        }
                        if (hide_field) {
                            error.element.closest(".row").slideUp(100);
                        }
                        else {
                            error.element.closest(".row").slideDown(100);
                        }
                    }

                    div = error.element.closest(".row").find(".error_message");
                    text = div.text();
                    div.text(text + (text.length > 0 ? "; " : "") + error.message);
                }
                return true;
            });

The following options can be passed to a new FormValidator object like so:
new FormValidator(form, options)

Error handling

create_dependency_error_message
Function(String locale, Array (of Object) dependency_errors) -> String
Can be used to customize what dependency error messages look like.
The default message is whatever error message is defined for the current locale at the key dependency.
Take a look at example 3 on the demos page to see how dependencies work.
dependency_error_classes
String (CSS class names, space separated)
Can be used to customize which CSS classes will be applied to the error target(s) of the fields the current field depends on (if any of the dependencies are invalid).
Default is "" (which means invalid fields won't be changed).
Take a look at example 3 on the demos page to see how dependencies work.
error_classes
String (CSS class names, space separated)
Defines what CSS classes will be applied to the error target(s) of an invalid field.
Default is "" (which means invalid fields won't be changed).
error_styles
String (CSS styles)
Defines what CSS styles will be applied to the error target(s) of an invalid field. Preferably, data-fv-error-classes should be used though.
Default is "" (which means invalid fields won't be changed).
error_messages
Object. It looks like this: 
                    {
                        locale1: {
                            error_type1: MESSAGE_1_1,
                            ...,
                            error_typeN: MESSAGE_1_N
                        },
                        locale2: {...}
                    }
where MESSAGE_X_Y is either a String or a Function(String locale, Object parameters) -> String. Can be used to override the default messages object (at FormValidator.error_messages). The object is merged into the predefined so customizing single message types (even for different locales) is possible.
The parameters argument depends on the error returned by the validator. The return value has at least these properties:
                    {
                        element: validated_field,
                        index: element_index (incl. optional fields),
                        index_of_type: index_of_validation_type,
                        name: validated_field.attr("data-fv-name"),
                        previous_name: name_of_previously_validated_field,
                        value: postprocessed_validated_field_value || preprocessed_validated_field_value || validated_field_value
                    }
Additional properties may be provided by the validator. Those could either properties describing the constraint attributes or "private" properties (that should be the case only if the validator is "private"...see validators for more details).
error_mode
FormValidator.ERROR_MODES (String)
Defines how error messages are generated in general. Can be either of these values:
  • FormValidator.ERROR_MODES.SIMPLE
    This mode transforms all error_message_types (returned from the validator) to the simplest corresponding error message. That means that the messages are independent of anyconstraint attributes. For example, if a field is validated as number and is invalid (no matter what combination of constraint attributes it has) the error_message_type will be "number" (even if it techniqually valid but violates constraints).
  • FormValidator.ERROR_MODES.NORMAL
    The error_message_type (returned from the validator) is untouched.
  • FormValidator.ERROR_MODES.DEFAULT (== FormValidator.ERROR_MODES.NORMAL) (this value is the default obviously)
error_output_mode
FormValidator.ERROR_OUTPUT_MODES (String)
Defines how error messages are shown. Can be either of these values:
  • FormValidator.ERROR_OUTPUT_MODES.BELOW
    Prints an error message below the according validated form field.
  • FormValidator.ERROR_OUTPUT_MODES.BOOTSTRAP_POPOVER
    Puts an error message into a bootstrap popover that shows when hovering over the according form field.
    Of course, this only works when Boostrap is included.
  • FormValidator.ERROR_OUTPUT_MODES.NONE
    The DOM will not be modified. The validate function returns an Array of Objects where each of those objects looks like this:
                                {
                                element: validated_field,
                                message: error_message,
                                required: validated_field.attr("data-fv-optional") !== "true",
                                type: validation_type,
                                validator: validator_function,
                                value: postprocessed_validated_field_value || preprocessed_validated_field_value || validated_field_value
                            }
  • FormValidator.ERROR_OUTPUT_MODES.DEFAULT (== FormValidator.ERROR_OUTPUT_MODES.NONE) (this value is the default obviously)
error_target_getter
Function(String validation_type, jQuery element, Number index) -> jQuery | String
Can be used to define the error target(s) for a validated field.
If a jQuery set is returned all the contained elements will be the targets.
If a String is returned it is parsed the same as if it was set using the data-fv-error-targets attribute. See target finding.

Validation handling

field_getter
Function(jQuery form) -> jQuery
Can be used to define the set of form fields that are relevant to the FormValidator.
locale
Define what locale is used. This property is relevant for:
  • creating error message
  • pre- and postprocessing values
optional_field_getter
Function(jQuery fields) -> jQuery
Can be used to define the subset of all fields (see field_getter) that contains all the fields that are optional. Those fields will only be validated if they are non-empty.
group
Function(jQuery fields) -> Array (of Array of jQuery)
Creates groups from all form fields. Can be used for advanced progress counting.
preprocessors
Object. It looks like this: 
                    {
                        validation_type1: PREPROCESSOR_1,
                        ...,
                        validation_typeN: PREPROCESSOR_N
                    }
where PREPROCESSOR_N is a Function(String value, jQuery element, String locale) -> String.
Can be used to override or define preprocessors. They are used modify the form field's value before the validation happens.
The predefined preprocessors currently exist for number and integer. Currently, for both types the comma will be interpreted as decimal points when the locale de is used.
postprocessors
Object. It looks like this: 
                    {
                        validation_type1: POSTPROCESSOR_1,
                        ...,
                        validation_typeN: POSTPROCESSOR_N
                    }
where POSTPROCESSOR_N is a Function(String value, jQuery element, String locale) -> String.
Can be used to override or define postprocessors. They are used modify the form field's value after the validation happened. The value is either the input's original or preprocessed value.
For example, a value could be auto-corrected i.e. the numeric string " 1.34 " would become "1.34" in the according field (when valid).
required_field_getter
Function(jQuery fields) -> jQuery
Can be used to define the subset of all fields (see field_getter) that contains all the fields that are required. Those fields will always be validated.
validation_options
Specify options for the validation:
  • Boolean all: if true force validation on all fields (even if they are declared optional)
  • Boolean apply_error_styles: if true add error_classes to and remove them from invalid elements' error targets (otherwise the targets won't be changed)
validators
Set of validators that correspond to a validation type (data-fv-validate).
They can be used to override existing validators or define new ones.
The predefined validators are:
  • checkbox
  • email
  • integer
  • number
  • phone
  • radio
  • select
  • text
The following validators come with the FormValidator. The list shows what values will be interpreted as valid.
  • checkbox
    a checked checkbox
  • email
    <ANYTHING+>@<ANYTHING_WITH_A_DOT+>
  • integer
    any integer x ∈ ℤ,
    Range can be defined with data-fv-min & data-fv-max (and data-fv-include-min & data-fv-include-max but that wouldn't make so much sense).
  • number
    any number x ∈ ℝ (that JavaScript can represent),
    Range can be defined with data-fv-min & data-fv-max and data-fv-include-min & data-fv-include-max.
  • phone
    any combination of "0123456789+-/() " (that's longer than 2 characters by default),
    As with text a minimum and maximum length can be defined using data-fv-min and data-fv-max.
  • radio
    a radio button of the element's group of radio buttons (= all with the same name attribute) must be checked
  • select
    the selected item's value is not null and at least 1 character long,
    As with text a minimum and maximum length can be defined using data-fv-min and data-fv-max.
  • text
    anything longer than 0 characters,
    A minimum and maximum length can be defined using data-fv-min and data-fv-max.

Dependencies can be used on any form field - let's pick one and call it F. The dependencies define what other form fields need to be valid before F can be valid (no matter the content).

It is possible to change F whenever its dependencies are "valid enough". With "valid enough" I mean that either

  • all of F's dependencies are fulfilled (onDependenciesValid) or
  • any of F's dependencies is fulfilled (onDependenciesInvalid).
Whether all or any is evaluated can be defined with the data-fv-dependency-mode attribute (either of ["all", "any"], default is all).

The change action can be anything if you use JavaScript.
However, usually you want to toggle the state of F on each event. So there is a list of predefined toggling changes. The naming for the actions is based on "do something whenever the dependencies are fulfilled - otherwise reverse whatever has been done before". Here we go:

  • fade *
    Fade the element in (on onDependenciesValid) or out (on onDependenciesInvalid).
    See jQuery's fadeIn and fadeOut.
  • opacity *
    Same as fade but the display style is not changed.
  • show *
    See jQuery's show and hide.
  • slide *
    See jQuery's slideDown and slideUp.
  • enable
    Change the disabled attribute (disabled on onDependenciesInvalid).
  • display
    The same as show but without animnation (changes the display style).
* The duration used for the animation is jQuery's default value (currently 400 ms). It is hardcoded though and will therefore not change automatically if jQuery changes it.

Cycle detection

There is none!
So you must make sure you do not define cyclic dependencies. Since dependencies are defined as a list of targets you could use a selector to find a dependency's element (or multiple). In that case you should be careful to match only the wanted elements.
Using the data-fv-name attribute should be easier and safer to use and is therefore recommended.

Instance methods

A FormValidator object has the following instance methods. Most of the functionality can be used equally by passing options to the FormValidator's constructor which is recommended for most use cases.
Function deregister_preprocessor(String validation_type) -> FormValidator chainable
Removes the preprocessor for the given validation_type from the set of preprocessors.
Function deregister_postprocessor(String validation_type) -> FormValidator chainable
Removes the postprocessor for the given validation_type from the set of postprocessors.
Function deregister_validator(String validation_type) -> FormValidator chainable
Removes the validator for the given validation_type from the set of validators.
Function get_progress(as_percentage=false) -> Object | Number
Calculate how much the form has been filled in.
Function register_preprocessor(String validation_type, Function preprocessor) -> FormValidator chainable
Adds the preprocessor for the given validation_type to the set of preprocessors.
The preprocessor parameter has this signature: Function(String value, jQuery element, String locale) -> String. In this function this refers to the preprocessor object so other preprocessors can easily be called from within each preprocessor.
Function register_postprocessor(String validation_type, Function postprocessor) -> FormValidator chainable
Adds the postprocessor for the given validation_type to the set of postprocessors.
The postprocessor parameter has this signature: Function(String value, jQuery element, String locale) -> String. In this function this refers to the postprocessor object so other preprocessors can easily be called from within each postprocessor.
Function register_validator(String validation_type, Function validator) -> FormValidator chainable
Adds the validator for the given validation_type to the set of validators.
The validator parameter has this signature: Function(String value, jQuery element) -> Boolean | String | Object. In this function this refers to the validator object so other validators can easily be called from within each validator.
Function set_error_target_getter(Function error_target_getter) -> FormValidator chainable
The error_target_getter parameter defines the error target for each validated element. The signature is Function(String validation_type, jQuery element, Number index) -> jQuery.
By default the form validator instance will check the data-fv-error-targets attribute.
Function set_field_getter(Function field_getter) -> FormValidator chainable
The field_getter parameter defines what elements (in the form) will be validated. The signature is Function(jQuery form) -> jQuery.
If you want the data-fv-ignore-children attribute to have an effect use it here.
By default all fields with a data-fv-validate attribute will be validated - excluding the ignored children.
Function set_optional_field_getter(Function optional_field_getter) -> FormValidator chainable
The optional_field_getter parameter defines what fields of all fields (defined by the field_getter) are optional. The signature is Function(jQuery fields) -> jQuery.
Function set_optional_field_getter(Function required_field_getter) -> FormValidator chainable
The required_field_getter parameter defines what fields of all fields (defined by the field_getter) are required (...mandatory == non-optional). The signature is Function(jQuery fields) -> jQuery.
Function validate(Object validation_options={all: false, apply_error_styles: true, focus_invalid: true}) -> Array (of Object)
Validates the form and returns a list of errors. Each error looks like this: 
                        {
                            element: validated_element (jQuery),
                            message: error_message (String),
                            required: is_required (Boolean),
                            type: validation_type (String),
                            validator: validator (Function),
                            value: value (String)
                        }
The validator and value properties are not present in dependency errors.

The validation_options has the following properties:
  • Boolean all: if true force validation on all fields (even if they are declared optional)
  • Boolean apply_error_styles: if true add error_classes to and remove them from invalid elements' error targets (otherwise the targets won't be changed)
  • Boolean focus_invalid: if true focus the first invalid element (otherwise the focus won't be changed)

Class methods

A FormValidator class has the following class methods.
Function FormValidator.new(jQuery form, Object options)
Returns a new instance of the FormValidator class. Right now using this method is equal to creating a new instance with new.
It may provide additional functionality in future versions so using this method is recommended.
data-fv-name
String (no space allowed)
Add a name property to the current element. It can be used for being identified as a target (see target finding) and for supplying a name variable in the error message.
data-fv-dependency-error-classes
String (CSS class names, space separated)
Can be used to customize which CSS classes will be applied to the error target(s) of a field with dependencies (if any of the dependencies are invalid).
Default is whatever is defined for the error_classes property.
data-fv-dependency-mode
String -> "all" or "any"
Defines whether all dependencies need to be fulfilled for the current element to be able to be valid or if one (any) is sufficient.
Default is "all".
For a more detailed explanation see dependencies.
data-fv-depends-on
String (selectors and/or target names, semicolon separated)
Defines what targets have to be valid for the current element to be valid (additionally to the actual validation). To find out how a target can be defined see target finding.
data-fv-error-classes
String (CSS class names, space separated)
Defines what CSS classes will be applied to the error target(s) of an invalid field.
Default is "" (which means invalid fields won't be changed).
data-fv-error-targets
String (selectors and/or target names, semicolon separated)
Defines a list of targets which will get the defined data-fv-error-classes if the current element is invalid. To find out how a target can be defined see target finding.
data-fv-group
String
Assigning the current element to a logical group - the value can be any string.
By assigning (potentially) all form fields to groups this defines a set of group for the form. This is used for progress counting.
It will not be used if the group option was given to the FormValidator's constructor.
To see how counting works see the get_progress method.
data-fv-ignore-children
String (any value)
This attribute can be used without a value. When present the default field_getter will ignore the elements' descendants. Therefore they won't be validated. The default field_getter is used unless you define your own (by passing the field_getter option to the constructor).
data-fv-include-max
Boolean
Define whether to include the defined data-fv-max or not.
data-fv-include-min
Boolean
Define whether to include the defined data-fv-min or not.
data-fv-max
Number
Define either the maximum value for an element with the validation type number or integer, or the maximum length for an element with the validation type phone, select, or text).
By default this value will be the greatest valid value. To exclude the maximum see data-fv-include-max.
data-fv-min
Number
Define either the minimum value for an element with the validation type number or integer, or the minimum length for an element with the validation type phone, select, or text).
By default this value will be the smallest valid value. To exclude the minimum see data-fv-include-min.
data-fv-optional
Boolean
When true the field will be skipped if it has an empty value, is a checkbox, or is a radio button.
data-fv-output-preprocessed
Boolean
If this value is true, there is a preprocessor, and there is no postprocessor for the element's type the element's value will be replaced to represent whatever was created by the preprocessor. This can be used for auto correcting numbers (i.e. remove spaces).
data-fv-postprocess
Boolean
Define whether to apply the postprocessor for the element's type to the element's value or not.
data-fv-preprocess
Boolean
Define whether to apply the preprocessor for the element's type to the element's value or not.
data-fv-real-time
String (any value)
This attribute can be used without a value. This attribute can be defined on the form element which is supposed to be validated. All descendants of the form with a data-fv-validate attribute will be validated on the fly whenever something changes (on change, click, and keyup).
data-fv-start
String (target)
Elements with this attribute will automatically trigger the validation of the form that corresponds to this value (which must be a valid target). To find out how a target can be defined see target finding.
data-fv-validate
String (validation type)
Define what validator to use for the current element. Possible values are:
  • checkbox
  • email
  • integer (also see data-fv-include-max, data-fv-include-min, data-fv-max, and data-fv-min)
  • number (also see data-fv-include-max, data-fv-include-min, data-fv-max, and data-fv-min)
  • phone
  • radio
  • select
  • text
For more details see validators.

Target finding

A list of targets is defined as semicolon separated String.

Each list element E is parsed like so (for a FormValidator F):

  1. try to find an element in the form of F where the attribute data-fv-name equals E
  2. otherwise E is interpreted as selector which will be looked for in the path to F (using jQuery's closest()), this makes it easy to affect container elements of E
  3. otherwise E is interpreted as selector which will be looked for in F
  4. otherwise E is interpreted as selector which will be looked for in the entire document

Note: you cannot use the ; character in the selector because it's used as the target delimiter.
